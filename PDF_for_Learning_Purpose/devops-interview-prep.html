☸ Kubernetes Core 10 Q ▶
01 What is a pod and why is it the smallest unit in Kubernetes? easy ▼

A pod is a wrapper around one or more containers that share the same network namespace, IP address, and storage volumes. Kubernetes doesn't schedule individual containers — it schedules pods.

Containers in the same pod communicate via localhost. They share the pod's lifecycle: if the pod dies, all containers in it die.

Why smallest? K8s needs a schedulable unit above a container — something that can encapsulate co-located processes (e.g., an app + a log sidecar) as a single unit of deployment, scaling, and networking. That unit is the pod.

In ApnaKart: each microservice ran as its own pod in its namespace. Datadog agent ran as a DaemonSet pod — one per node, sharing the node's network namespace.
02 Difference between Deployment, StatefulSet, and DaemonSet? medium ▼

Deployment — Stateless workloads. Pods are interchangeable, no stable identity. Rolling updates, scaling, rollbacks. Used for APIs, web apps. In ApnaKart: all 10 Spring Boot microservices ran as Deployments.

StatefulSet — Stateful workloads needing stable identity. Pods get ordered names (pod-0, pod-1), stable DNS (pod-0.svc.ns.svc.cluster.local), persistent volume per pod. Ordered deploy/scale/delete. Used for MySQL, Cassandra, Kafka.

DaemonSet — Runs exactly one pod per node (or matching nodes). Pod follows node lifecycle. Used for node-level agents. In ApnaKart: Datadog agent deployed as DaemonSet — ensured every EKS node had APM collection running.

Interview tip: always tie DaemonSet to Datadog from your project. It shows real-world usage, not textbook knowledge.
03 What is CrashLoopBackOff and how do you debug it? hard ▼

CrashLoopBackOff = pod starts, container crashes, K8s waits before restarting (backoff: 10s → 20s → 40s → 80s → 160s → max 5min), then retries. It's not a crash type — it's a restart loop state.

Debug steps:

1kubectl describe pod <pod> -n <ns> → check Last State and Events section
2kubectl logs <pod> --previous → logs from the crashed container
3Check exit code: 137=OOMKilled, 1=app error, 139=segfault, 0=liveness probe killing healthy container
4Verify env vars, ConfigMaps, Secrets are correctly mounted
5Check if liveness probe is too aggressive (low initialDelaySeconds)
6Check if dependent services (DB, API) are reachable
7Test container standalone: kubectl run debug --image=<img> -- sleep 3600 then exec in
04 What happens when a pod goes into Evicted state? medium ▼

Kubelet evicts pods when a node is under resource pressure (memory, disk, PIDs). The pod is terminated immediately — not gracefully restarted on the same node.

The evicted pod object remains in etcd with status: Evicted until manually deleted or garbage collected. The scheduler creates a new replacement pod on a different node (if it was managed by a Deployment/ReplicaSet).

Key difference from CrashLoopBackOff: Eviction is node-driven (resource pressure), not app-driven (process crash).

kubectl get pods --field-selector status.phase=Failed -A | grep Evicted
kubectl delete pods --field-selector status.phase=Failed -n <ns>
05 Difference between OOMKilled and Evicted? medium ▼

OOMKilled (exit code 137):

  • Container exceeded its memory limit
  • Linux kernel OOM killer terminates that specific container
  • Pod may restart (depending on restartPolicy)
  • Only the offending container is killed, not the whole pod
  • Seen in kubectl describe pod → Last State: OOMKilled

Evicted:

  • Entire pod removed by kubelet due to node-level resource exhaustion
  • Not caused by a single container's limit — node is running out of memory/disk globally
  • Pod is not restarted on same node — scheduler picks a new node
  • Pod status becomes Failed with reason Evicted
06 How does Kubernetes decide which pod to evict? hard ▼

Kubelet evicts in this priority order (lowest first):

  • BestEffort pods first — no requests or limits defined
  • Burstable pods using more than their requests — ordered by how much over their request they are
  • Guaranteed pods last — only evicted if system-level processes need resources

Additionally, Pod Priority Class matters — lower priorityClassName value = evicted first within the same QoS tier.

Kubelet also considers hard eviction thresholds (e.g., memory.available < 100Mi) vs soft eviction (with grace period).

07 What are QoS classes — BestEffort, Burstable, Guaranteed? medium ▼

Guaranteed — Highest protection. Requires: requests == limits for ALL containers, both CPU and memory.

Burstable — At least one container has requests set, but requests ≠ limits. Or only requests defined. Middle tier.

BestEffort — No requests, no limits for any container. Zero protection. First to be evicted. Do NOT run production workloads as BestEffort.

In ApnaKart: all microservices had explicit requests and limits defined (Burstable or Guaranteed). This contributed to 99.9% availability — pods weren't randomly evicted.
resources:
  requests:
    memory: "256Mi"
    cpu: "250m"
  limits:
    memory: "512Mi"
    cpu: "500m"   # Burstable (requests != limits)
08 How do liveness and readiness probes work? medium ▼

Liveness probe — "Is this container alive?" If it fails, kubelet kills the container and restarts it (subject to restartPolicy). Use for detecting deadlocks where the process is running but stuck.

Readiness probe — "Is this container ready to serve traffic?" If it fails, the pod is removed from the Service endpoints. Traffic stops. Pod is NOT restarted. Use for slow startup or temporary unavailability.

Startup probe — Added in K8s 1.16. Delays liveness checks during slow boot. Once startup passes, liveness takes over. Prevents liveness from killing a slow-starting container.

livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
09 What happens if readiness probe fails? easy ▼

Pod is removed from the Service Endpoints list. kube-proxy updates iptables rules. New requests no longer route to this pod.

The pod itself keeps running — kubelet does NOT restart it. Once the probe starts passing again, the pod is re-added to endpoints automatically.

Common scenario: During rolling update, new pods fail readiness → stay out of rotation → old pods keep serving → zero downtime. This is why readiness probes are critical for zero-downtime deployments.

kubectl get endpoints <service-name>  # See which pods are in rotation

🌐 Networking & DNS 7 Q ▶
11 How does DNS work inside Kubernetes? medium ▼

Every pod's /etc/resolv.conf is configured to point to CoreDNS (ClusterIP, usually 10.96.0.10).

Service DNS record: <service>.<namespace>.svc.cluster.local

Within same namespace: just <service-name> resolves (search domains fill the rest).

Pod DNS: <pod-ip-dashes>.<namespace>.pod.cluster.local (e.g., 10-0-1-5.default.pod.cluster.local)

cat /etc/resolv.conf  # inside a pod
# nameserver 10.96.0.10
# search default.svc.cluster.local svc.cluster.local cluster.local
12 What is CoreDNS and how do you troubleshoot it? hard ▼

CoreDNS is a flexible DNS server deployed as a Deployment in kube-system namespace. It replaced kube-dns in K8s 1.13+. Configured via a Corefile (ConfigMap).

1kubectl get pods -n kube-system | grep coredns — are pods Running?
2kubectl logs -n kube-system <coredns-pod> — look for NXDOMAIN loops, SERVFAIL, OOM
3kubectl get cm coredns -n kube-system -o yaml — check Corefile config
4Check if CoreDNS has resource limits and is getting OOMKilled (common under high DNS traffic)
5Enable log plugin in Corefile for debug logging (disable after)
13 How do you debug DNS issues in a pod? hard ▼ 
kubectl run dns-debug --image=busybox:1.28 --rm -it -- sh

# Inside the pod:
cat /etc/resolv.conf                                    # check nameserver
nslookup kubernetes.default                             # test cluster DNS
nslookup <svc>.<namespace>.svc.cluster.local           # test service DNS
nslookup google.com                                     # test upstream DNS
dig @10.96.0.10 <service>.<ns>.svc.cluster.local

Common findings:

  • Wrong nameserver → CoreDNS ClusterIP changed after restart
  • NXDOMAIN for service → service name/namespace typo, or service doesn't exist
  • Timeout → CoreDNS pod down, or NetworkPolicy blocking UDP 53
  • Slow resolution → high ndots value causing multiple lookup attempts
14 What is ClusterIP, NodePort, LoadBalancer? easy ▼

ClusterIP — Default. Virtual IP only reachable within the cluster. Used for internal service-to-service communication. Cannot be accessed from outside.

NodePort — Exposes service on every node's IP at a static port (30000–32767). External traffic: NodeIP:NodePort. NodePort wraps ClusterIP — both still work.

LoadBalancer — Provisions a cloud load balancer (AWS ALB/NLB on EKS). External traffic → Cloud LB → NodePort → ClusterIP → Pod. Wraps NodePort wraps ClusterIP.

In ApnaKart on EKS: used AWS ALB (via AWS Load Balancer Controller) as LoadBalancer type, with SSL termination at ALB level and CloudFront in front for CDN caching.
15 What is CNI and how does networking happen between pods? hard ▼

CNI (Container Network Interface) = standard interface for K8s to plug in network providers. kubelet calls CNI plugin when pod is created to set up pod's network interface.

AWS VPC CNI (used on EKS): Each pod gets a real VPC IP (from the node's ENI secondary IPs). No overlay network — pod traffic routed directly via VPC routing tables. Same as EC2 instances. This gives low latency and native VPC security group support.

Other CNI plugins: Calico (eBPF, NetworkPolicy support), Flannel (VXLAN overlay, simpler), Weave (mesh).

Pod-to-pod same node: Via virtual ethernet (veth) pair → bridge.

Pod-to-pod cross node: Via VPC routing (AWS CNI) or VXLAN tunnel (Flannel).

16 How do you debug intermittent packet loss? hard ▼
1Check CNI pod health: kubectl get pods -n kube-system — any CNI pods restarting?
2Check MTU mismatch: common with VPN/overlay networks. AWS CNI default MTU is 9001 (jumbo frames). Mismatch causes silent drops.
3tcpdump -i any -n host <pod-ip> inside node — see if packets arrive but not responded
4Check iptables: iptables -L -n -v | grep DROP — unexpected drops?
5Check kube-proxy logs — iptables rule corruption is rare but happens
6Check node CPU steal time — packet drops under load on shared infra
7Use ping with flood test: ping -f -c 1000 <dest> and check loss %
17 How does service discovery work in Kubernetes? easy ▼

Two mechanisms:

DNS-based (primary): CoreDNS auto-creates DNS records for every Service. Pod resolves order-service → CoreDNS → returns ClusterIP. Preferred method.

Environment variables (legacy): K8s injects env vars into pods for services that existed at pod creation time (e.g., ORDER_SERVICE_HOST, ORDER_SERVICE_PORT). Problem: only services created before the pod are available. Not recommended for new apps.

In ApnaKart: all 10 microservices discovered each other via DNS short names within the same namespace. Cross-namespace calls used full FQDN.
☁ AWS / Cloud 6 Q ▶
18 Difference between Internet Gateway and NAT Gateway? medium ▼

Internet Gateway (IGW):

  • Allows bidirectional traffic between VPC resources and the internet
  • Resources need a public IP or Elastic IP
  • Attached to the VPC, not a subnet
  • Stateful — no extra config for return traffic

NAT Gateway:

  • Allows outbound-only internet access from private subnets
  • Resources keep their private IPs — NAT translates to NAT's public IP
  • Lives in a public subnet with its own Elastic IP
  • Inbound connections from internet are not possible
In ApnaKart: EKS worker nodes in private subnets → used NAT GW for pulling Docker images from DockerHub/ECR. ALB in public subnet handled inbound traffic.
19 Why is NAT Gateway placed in a public subnet? medium ▼

NAT Gateway needs to forward outbound traffic to the internet — so it itself needs internet access. A public subnet is a subnet whose route table has a 0.0.0.0/0 → IGW route.

NAT GW is assigned an Elastic IP (static public IP). When private instances send traffic through NAT GW, it uses this EIP as the source. The IGW then forwards that traffic to the internet.

Traffic flow: Private EC2 → Route table (0.0.0.0/0 → NAT GW) → NAT GW → Public subnet route table (0.0.0.0/0 → IGW) → IGW → Internet

20 What is VPC and how does routing work? easy ▼

VPC (Virtual Private Cloud) = logically isolated network in AWS. Define your own CIDR block (e.g., 10.0.0.0/16). Contains subnets across AZs.

How routing works:

  • Each subnet is associated with exactly one route table
  • Route table has ordered rules: destination CIDR → target
  • Local route (10.0.0.0/16 → local) always present — covers intra-VPC traffic
  • For internet: add 0.0.0.0/0 → IGW (public) or 0.0.0.0/0 → NAT GW (private)
  • Most specific route wins (longest prefix match)
In ApnaKart Terraform: created separate route tables for public (→IGW) and private (→NAT GW) subnets. This was part of the VPC module.
21 Difference between Security Groups and NACLs? medium ▼

Security Groups (SG):

  • Stateful — return traffic automatically allowed
  • Applied to instances/ENIs (resource level)
  • Allow rules only — no deny rules
  • All rules evaluated together (no order)

Network ACLs (NACL):

  • Stateless — must explicitly allow both inbound AND outbound return traffic
  • Applied at subnet boundary
  • Allow and Deny rules — evaluated in order (lowest rule number first)
  • First matching rule wins — rule 100 ALLOW overrides rule 200 DENY

Evaluation order: NACL checked first (at subnet boundary), then SG (at instance level). Both must allow traffic for it to pass.

22 What happens if route table is misconfigured? medium ▼

Common misconfiguration scenarios:

  • Missing 0.0.0.0/0 → IGW in public subnet: Public EC2 can't reach internet, ALB can't receive external traffic
  • Missing 0.0.0.0/0 → NAT GW in private subnet: Private EC2 can't reach internet (can't pull images, packages)
  • NAT GW in wrong subnet: NAT GW placed in a private subnet — it has no internet access itself, so all outbound traffic fails
  • Wrong subnet association: Private subnet accidentally uses public route table → security risk; resources get direct internet exposure
  • VPC Peering missing return route: One-way connectivity, connection timeouts

Debug: VPC → Route Tables → Check routes and subnet associations. Use VPC Reachability Analyzer for automated path verification.

23 How does traffic flow from private subnet to internet? easy ▼

Full path with each component:

Private EC2 (10.0.2.5)
  → Private subnet route table: 0.0.0.0/0 → nat-xxxxx
  → NAT Gateway (in public subnet, EIP: 54.x.x.x)
  → Public subnet route table: 0.0.0.0/0 → igw-xxxxx
  → Internet Gateway
  → Internet (src IP appears as NAT GW's EIP)

Return path is the reverse — but because NAT is stateful, it knows to send return traffic back to the originating private IP.

⚙ Resources & Performance 5 Q ▶
31 What are requests and limits? easy ▼

Requests = minimum guaranteed resources. Used by the scheduler to decide if a node has enough capacity to place the pod. Node must have this much available. Container is guaranteed this amount.

Limits = maximum allowed. If a container exceeds its memory limit → OOMKilled (immediate termination). If it exceeds its CPU limit → throttled (slowed, not killed).

CPU is measured in millicores: 250m = 0.25 CPU core. Memory in Mi/Gi.

In ApnaKart: all 10 microservices had requests and limits explicitly set in Kustomize overlays, different per environment (dev had lower limits, prod had higher). This contributed to zero unexpected evictions in production.
32 What happens if limits are not defined? medium ▼

Pod gets BestEffort QoS — first to be evicted under pressure. Extremely dangerous in production.

Container can consume all available node resources, starving other pods on the same node. One memory leak can bring down multiple services.

For CPU only (no memory limit): Pod is Burstable QoS. Can use all spare CPU but gets throttled when competing. Memory remains unbounded — still risky.

Best practice: Always set both requests and limits. Use LimitRange in each namespace to enforce defaults and maximums automatically.

33 What causes OOMKilled and how to prevent memory issues? medium ▼

Causes:

  • Memory leak — objects accumulate in heap, GC can't free them
  • Memory limit set too low for the actual workload
  • JVM heap (-Xmx) exceeds container memory limit — JVM doesn't respect cgroup limits by default (pre-Java 11)
  • Sudden traffic spike causing heap growth beyond limit
  • Unbounded caches (no eviction policy)

Prevention:

  • Set -Xmx to ~75% of container memory limit (leave headroom for non-heap)
  • Java 11+: use -XX:+UseContainerSupport — JVM auto-detects cgroup limits
  • Monitor with Datadog/Prometheus: alert at >80% memory usage
  • Load test to find realistic baseline before setting limits
  • Implement cache eviction (LRU, TTL-based)
34 How do you monitor resource usage? easy ▼

kubectl level: kubectl top pods, kubectl top nodes (requires metrics-server)

Prometheus + Grafana: Scrapes kubelet/cAdvisor metrics. Key metrics: container_memory_usage_bytes, container_cpu_usage_seconds_total, container_cpu_throttled_seconds_total

In ApnaKart: used Datadog APM as DaemonSet for container-level resource metrics, infrastructure metrics, and distributed traces. Set alerts on memory usage >80% and CPU throttle rate >20%.

VPA (Vertical Pod Autoscaler): In recommendation mode, analyzes actual usage and suggests right-sized requests/limits. Non-disruptive way to right-size.

35 What is HPA and how does autoscaling work? medium ▼

HPA (Horizontal Pod Autoscaler) scales pods in/out based on observed metrics.

Internals:

  • Metrics Server scrapes pod metrics every 15s from kubelet
  • HPA controller checks metrics every 30s (default)
  • Formula: desiredReplicas = ceil(currentReplicas × (currentMetric / targetMetric))
  • Scale-up: immediate (no delay)
  • Scale-down: 5min stabilization window (prevents flapping)
  • Respects minReplicas and maxReplicas bounds
In ApnaKart: HPA configured on all microservices targeting 70% CPU. Scaled from 2 to 10 replicas. Combined with ALB's health checks for zero-downtime scaling. This contributed to 99.9% availability under load.
kubectl get hpa -n production
kubectl describe hpa <name>  # shows current/target metrics

37 What is blue-green deployment? easy ▼

Two identical production environments: Blue (current live) and Green (new version).

Deploy new version to Green while Blue keeps serving traffic. Run smoke tests on Green. When ready: switch traffic (LB/DNS/Service selector change) to Green. Blue stays as instant rollback.

Pros: Zero downtime, instant rollback, easy testing in production-like environment.

Cons: Double infrastructure cost during deployment. DB schema changes are tricky (must be backward compatible).

In Kubernetes: Two Deployments (blue/green), one Service. Change Service selector label from version: blue to version: green to switch traffic.

kubectl patch service myapp -p '{"spec":{"selector":{"version":"green"}}}'
38 What is rolling update in Kubernetes? easy ▼

Default Deployment update strategy. Gradually replaces old pods with new ones — never takes all pods down at once.

Controlled by two parameters:

  • maxSurge: how many extra pods can exist above desired count (e.g., 25% = can temporarily have 5 pods for a 4-replica deployment)
  • maxUnavailable: how many pods can be unavailable during update (e.g., 25% = up to 1 pod can be down)

Zero-downtime requires: readiness probes correctly configured. If new pod fails readiness, rolling update pauses. Old pods keep serving.

kubectl rollout status deployment/myapp -n production
kubectl rollout history deployment/myapp
In ApnaKart: maxSurge=1, maxUnavailable=0 for prod — always had full capacity during update. Readiness probes ensured traffic only went to healthy pods.
39 How do you rollback a failed deployment? medium ▼

Kubernetes native:

kubectl rollout undo deployment/<name> -n <ns>
kubectl rollout undo deployment/<name> --to-revision=<N>
kubectl rollout history deployment/<name>         # see revisions

In Jenkins pipeline: Trigger the previous successful build. Or add a ROLLBACK parameter that deploys the last known-good image tag.

GitOps approach: Revert the commit that changed the image tag/config in Git → ArgoCD/Flux automatically deploys the previous state.

After rollback: Always check the logs, metrics, and conduct a post-mortem before the next deploy attempt. Fix the root cause — not just the rollback.

🧠 Pro-Level / Deep Internals 5 Q ▶
41 How does the Kubernetes scheduler work internally? hard ▼

Two-phase process:

Phase 1 — Filtering (Predicates): Eliminate nodes that can't run the pod.

  • Insufficient CPU/memory requests
  • Taint/toleration mismatch
  • NodeSelector / nodeAffinity constraints
  • Volume zone requirements
  • Port conflicts

Phase 2 — Scoring (Priorities): Rank feasible nodes. Key scoring functions:

  • LeastRequestedPriority: prefer less-loaded nodes (spread)
  • ImageLocalityPriority: prefer nodes that already have the container image
  • InterPodAffinityPriority: prefer nodes where preferred pods are co-located

Binding: Scheduler writes nodeName to the pod spec via API server. Kubelet on that node picks it up via watch.

42 What happens inside kubelet when a pod is created? hard ▼
1kubelet watches API server for pods assigned to its node (spec.nodeName == thisNode)
2Calls CRI (containerd/CRI-O) to pull container images
3Calls CNI plugin to create network namespace and assign pod IP, set up veth pair
4Mounts volumes: ConfigMaps, Secrets (projected volumes), PersistentVolumeClaims via CSI
5Starts init containers in order — waits for each to complete successfully before next
6Starts application containers in parallel (unless dependencies defined)
7Starts liveness/readiness/startup probes as goroutines
8Reports pod status (Running, container states) back to API server via status update
43 How does service routing work at iptables level? hard ▼

kube-proxy watches Services and Endpoints objects. For each Service, it creates iptables rules in the PREROUTING and OUTPUT chains.

Packet flow for ClusterIP → Pod:

Packet arrives with dst: 10.100.0.5:80 (ClusterIP)
  → PREROUTING chain → KUBE-SERVICES chain
  → Match: dst 10.100.0.5:80 → jump to KUBE-SVC-xxxxx
  → KUBE-SVC-xxxxx: randomly pick one of N endpoints (statistic --probability)
  → KUBE-SEP-yyy1: DNAT dst to 10.0.1.4:8080 (pod IP)
  → Packet forwarded to pod
  → Return packet: conntrack table reverses DNAT automatically

IPVS mode (alternative): Uses kernel hash tables instead of linear iptables chains. O(1) lookup vs O(n). Recommended for clusters with >1000 services.

44 What happens when a node goes NotReady? hard ▼
1Kubelet sends heartbeats to API server every 10s. node-monitor-grace-period = 40s. After 40s silence → node marked Unknown/NotReady
2Node controller adds taint: node.kubernetes.io/not-ready:NoExecute and node.kubernetes.io/unreachable:NoExecute
3Pods with no toleration for this taint: evicted after tolerationSeconds (default: 300s = 5min)
4DaemonSet pods: tolerate NotReady by default — they stay (or are recreated on recovery)
5Deployment pods: rescheduled on healthy nodes after 5min

Debug:

kubectl describe node <node>   # check Conditions section
ssh <node> && journalctl -u kubelet -n 100  # kubelet logs

Common causes: OOM on node, disk full, network partition, kubelet process died, Docker/containerd crashed.

45 Kubernetes control plane components — what does each do? medium ▼

API Server (kube-apiserver): Single entry point for all K8s operations. Validates and processes REST requests. All components communicate through it — nothing talks directly.

etcd: Distributed key-value store. Single source of truth for all cluster state. Highly available (3 or 5 node raft consensus). Everything in K8s is an object stored in etcd.

Scheduler (kube-scheduler): Watches for unscheduled pods. Selects best node via filter + score. Writes nodeName to pod spec.

Controller Manager: Runs all controllers as goroutines: ReplicaSet controller, Deployment controller, Node controller, Job controller, etc. Each reconciles actual state → desired state.

Cloud Controller Manager: On EKS: handles AWS-specific operations — LoadBalancer provisioning, node cloud labels, route management.

46 What is Exit Code 137? rapid ▼

Exit 137 = 128 + 9 = OOMKilled. Signal 9 = SIGKILL. Container was killed by the Linux OOM killer because it exceeded its memory limit.

Always check: kubectl describe pod → Last State: OOMKilled. Fix: increase memory limit or fix memory leak.

Other useful exit codes: 1 = general error, 126 = command not executable, 127 = command not found, 139 = segfault (SIGSEGV), 143 = graceful SIGTERM

47 What is the default DNS policy in Kubernetes? rapid ▼

ClusterFirst — cluster DNS (CoreDNS) is tried first. If hostname matches cluster domain suffix, it's resolved by CoreDNS. Otherwise, forwarded to upstream DNS (node's configured nameserver).

Other options: Default (use node's DNS, no cluster DNS), None (custom DNS via dnsConfig), ClusterFirstWithHostNet (for hostNetwork pods).

48 Can a pod move to another node automatically? rapid ▼

No — pods are not migrated. A pod is permanently bound to a node once scheduled.

What actually happens: the pod is deleted (eviction, node failure, etc.) and the Deployment/ReplicaSet controller creates a new pod. The new pod goes through scheduling again and may land on a different node. Different pod UID, different IP.

This is why stateless apps work well in K8s — any pod can be killed and recreated. StatefulSets use PVCs with ReadWriteOnce to persist data across pod recreations.

49 Difference between pod restart and pod recreation? rapid ▼

Restart = same pod object, container inside is killed and restarted by kubelet. Same pod IP, same node, same pod UID. restartCount increments. Happens on: liveness probe failure, OOMKill, app crash.

Recreation = pod object is deleted, new pod object is created by the controller (Deployment, ReplicaSet). New pod UID, potentially new node, new IP. Happens on: pod eviction, node failure, manual deletion, rolling update.

Check with: kubectl get pod <name> -o yaml | grep uid — if UID changes, it was recreated.

50 What is the role of kube-proxy? rapid ▼

kube-proxy runs on every node. It watches the API server for Service and Endpoint changes. When a new Service is created, kube-proxy programs the node's kernel (iptables or IPVS) to implement service routing.

Critical misconception: kube-proxy does NOT proxy actual network traffic. It only sets up kernel-level rules. The packet goes kernel → iptables → pod directly. kube-proxy is a rule installer, not a traffic forwarder.

Modes: iptables (default), IPVS (better for scale), userspace (legacy, avoid).
